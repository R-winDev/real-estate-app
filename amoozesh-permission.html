🐧 آموزش Linux File Permissions

از صفر شروع می‌کنیم. اگه تا حالا لینوکس کار نکردی، نگران نباش — همه چی رو با مثال از پروژه خودمون توضیح می‌دم.

📌 مقدمه: چرا باید این رو یاد بگیرم؟

توی Docker پروژه، فایل‌ها روی لینوکس اجرا می‌شن (حتی اگه خودت روی ویندوز کار می‌کنی). هر فایل توی لینوکس یه برچسب دسترسی داره که مشخص می‌کنه:

  • چه کسی می‌تونه این فایل رو بخونه؟
  • چه کسی می‌تونه توش تغییر بده (بنویسه)؟
  • چه کسی می‌تونه اجراش کنه؟

اگه این رو درست نفهمی، دو تا مشکل پیش میاد:

  1. Laravel ارور Permission denied برای storage/ می‌ده
  2. بدون دلیل فایل‌ها رو chmod 777 می‌کنی و سرورتو خطرناک می‌کنی

🔍 قدم ۱: اول ببینیم فایل‌هامون چه وضعیتی دارن

📋 دستور: ls -la

این دستور فایل‌ها رو با جزئیات نشون می‌ده. ls یعنی list، -l یعنی long format (جزئیات کامل)، -a یعنی all (فایل‌های مخفی رو هم نشون بده).

$ docker compose exec php ls -la resources/views/components/

🌸 خروجی:

total 68 drwxr-xr-x 2 www-data www-data 4096 Jul 5 09:29 . drwxr-xr-x 7 www-data www-data 4096 Jul 4 13:57 .. -rw-r--r-- 1 www-data www-data 1200 Jul 5 09:29 alert.blade.php -rw-r--r-- 1 www-data www-data 1035 Jul 5 06:43 property-card.blade.php -rw-r--r-- 1 www-data www-data 3045 Jul 4 13:57 application-logo.blade.php

💡 چرا از docker compose exec php استفاده می‌کنیم؟

چون خودمون روی ویندوز هستیم، فایل‌ها توی container لینوکسی (Debian) زندگی می‌کنن. برای دیدن permission واقعی باید بریم توی container. docker compose exec php یعنی: "برو توی container که اسمش php هست و اونجا دستور رو اجرا کن".

📋 دستور: stat (جزئیات کامل‌تر)

stat خیلی بیشتر از ls -la جزئیات نشون می‌ده — سایز دقیق، زمان آخرین تغییر، permission به دو فرمت مختلف (octal و symbolic)، و...

$ docker compose exec php stat resources/views/components/property-card.blade.php

🌸 خروجی:

File: resources/views/components/property-card.blade.php Size: 1035 Blocks: 8 IO Block: 4096 regular file Device: 8,48 Inode: 46130 Links: 1 Access: (0644/-rw-r--r--) Uid: ( 1000/www-data) Gid: ( 1000/www-data) Access: 2026-07-05 06:43:19 Modify: 2026-07-05 06:43:19 Change: 2026-07-05 06:43:19

📌 خط طلایی: Access: (0644/-rw-r--r--)

اینجا دو شکل از همون permission رو می‌بینی:

  • 0644 → فرمت octal (عددی)
  • -rw-r--r-- → فرمت symbolic (حرفی)

توی این آموزش یاد می‌گیری که چطور این دو رو به هم تبدیل کنی.

🧩 قدم ۲: الفبای rwx — هر کاراکتر یعنی چی؟

بیا اول permission یه فایل رو بشکافیم. مثلاً -rw-r--r-- رو تکه‌تکه می‌کنیم:

┌─┌───┐┌───┐┌───┐ - rw- r-- r-- │ │ │ │ │ │ │ └── Others (سایرین) │ │ └──────── Group (گروه) │ └────────────── Owner (مالک) └────────────────── نوع فایل
موقعیتنام بلوکچند حرف؟یعنی چی؟
کاراکتر ۱ نوع فایل ۱ تا - = فایل معمولی (مثل property-card.blade.php)
d = دایرکتوری (مثل components/)
l = لینک (shortcut) — مثلاً storage → storage/app/public
کاراکتر ۲-۴ Owner ۳ تا دسترسی‌های صاحب فایل. توی پروژه ما، همه فایل‌ها مال www-data هستن.
کاراکتر ۵-۷ Group ۳ تا دسترسی‌های گروه. هر کاربر لینوکس عضو یه گروهه. اگه گروه با گروه فایل یکی باشه، این دسترسی اعمال می‌شه.
کاراکتر ۸-۱۰ Others ۳ تا دسترسی بقیه مردم دنیا — هر کسی که نه owner هست و نه توی groupe. مثلاً کاربر ali که می‌خواد یه فایل از سرور شما بخونه.

حالا خود r و w و x چی هستن؟

حرفمخففمعادل عددیروی فایل یعنیروی دایرکتوری یعنی
r read ۴ می‌تونی فایل رو باز کنی و بخونی می‌تونی فهرست فایل‌های داخلش رو ببینی (ls dir/)
w write ۲ می‌تونی فایل رو تغییر بدی یا پاکش کنی می‌تونی توش فایل جدید بسازی، پاک کنی،改名 کنی
x execute ۱ می‌تونی فایل رو اجرا کنی (اگه script یا برنامه باشه) می‌تونی بری توی دایرکتوری (cd dir/)
- — ۰ این دسترسی داده نشده

💡 نکته خیلی مهم - x توی دایرکتوری:

توی دایرکتوری، x یعنی اجازه عبور. اگه یه دایرکتوری r-- داشته باشه (فقط read، بدون x)، می‌تونی اسم فایل‌هاش رو ببینی ولی نمی‌تونی واردش بشی یا فایل‌هاش رو بخونی! به همین خاطر همه دایرکتوری‌ها معمولاً x دارن.

🧮 قدم ۳: از rwx به عدد Octal — ریاضی که حتی از دبستان هم راحت‌تره

هر کدوم از حروف r, w, x یه عدد دارن. برای تبدیل به عدد octal کافیه هر بلوک سه‌تایی رو جمع بزنی:

r = ۴     w = ۲     x = ۱     - = ۰

✅ مثال از فایل خودمون: -rw-r--r--

بلوکدسترسیمحاسبه (جمع اعداد)عدد Octal
Owner rw- r(4) + w(2) + -(0) = 6 ۶
Group r-- r(4) + -(0) + -(0) = 4 ۴
Others r-- r(4) + -(0) + -(0) = 4 ۴

⇨ ۶۴۴ = rw- + r-- + r--

✅ مثال از دایرکتوری components/: drwxr-xr-x

دقت کن اولش d هست (یعنی دایرکتوری). بعدش سه بلوک میاد:

Owner
rwx
4+2+1 = ۷
Group
r-x
4+0+1 = ۵
Others
r-x
4+0+1 = ۵

⇨ ۷۵۵ = rwx + r-x + r-x

💡 روش میان‌بر برای تبدیل:

جای r/w/x رو با عددش بذار و جمع کن:

rw- → 4+2+0 = ۶   |   r-x → 4+0+1 = ۵   |   rwx → 4+2+1 = ۷   |   --- → 0+0+0 = ۰

❓ قدم ۴: چرا ۶۴۴ برای فایل‌ها و ۷۵۵ برای دایرکتوری‌ها؟

حالا که می‌دونی عددها چطور کار می‌کنن، بیا ببینیم چرا این دو تا عدد استاندارد هستن:

📄 فایل — ۶۴۴

644 = rw-r--r--

  • ✓ Owner (شما): می‌تونی بخونی و تغییر بدی
  • ✓ Group: فقط می‌تونه بخونه (نتونه خرابش کنه)
  • ✓ Others: فقط می‌تونن بخونن (مثل مرورگر)
  • ✓ x وجود نداره چون فایل معمولی نیازی به اجرا نداره

مثال از پروژه: property-card.blade.php رو همه می‌تونن بخونن (nginx نیاز داره بفرسته به مرورگر) ولی فقط ما می‌تونیم تغییرش بدیم.

📁 دایرکتوری — ۷۵۵

755 = rwxr-xr-x

  • ✓ Owner: دسترسی کامل (بخوانه، بنویسه، وارده بشه)
  • ✓ Group: می‌تونه وارد بشه و فایل‌ها رو ببینه ولی نتونه فایل جدید بسازه یا حذف کنه
  • ✓ Others: همینطور
  • ✓ x اجباریست وگرنه نمی‌تونی cd کنی!

مثال از پروژه: دایرکتوری components/ — nginx باید بتونه واردش بشه تا فایل‌های داخلش رو بفرسته به PHP.

⚡ قدم ۵: تست عملی — چیزایی رو که یاد گرفتی امتحان کن

حالا که فهمیدی عددها چطور کار می‌کنن، بیا یه تست انجام بدیم. می‌خوایم ببینیم اگه یه فایل رو chmod 777 کنیم، چه شکلی میشه و بعد برگردونیمش به ۶۴۴.

مرحله ۱: اول وضعیت فعلی رو می‌بینیم

$ docker compose exec php stat resources/views/components/property-card.blade.php
Access: (0644/-rw-r--r--) Uid: ( 1000/www-data) Gid: ( 1000/www-data)

فعلاً ۶۴۴ هست (rw-r--r--). بیاین عوضش کنیم.

مرحله ۲: می‌زنیمش ۷۷۷ (فقط برای تست!)

$ docker compose exec php chmod 777 resources/views/components/property-card.blade.php
$ docker compose exec php ls -la resources/views/components/property-card.blade.php
-rwxrwxrwx 1 www-data www-data 1035 Jul 5 06:43 property-card.blade.php

ببین چقدر تغییر کرد!

  • قبلاً: rw-r--r-- (سه تا - به جای x)
  • حالا: rwxrwxrwx — هر سه بلوک x گرفتن!

یعنی هر کسی روی سرور می‌تونه این فایل رو اجرا کنه (که danger هست چون اصلاً نیازی نیست).

مرحله ۳: برمی‌گردونیمش به حالت امن

$ docker compose exec php chmod 644 resources/views/components/property-card.blade.php
$ docker compose exec php ls -la resources/views/components/property-card.blade.php
-rw-r--r-- 1 www-data www-data 1035 Jul 5 06:43 property-card.blade.php

برگشت به حالت عادی. ✓

💡 حواست باشه: موقع اجرای این دستورات، در حال حاظر توی container با کاربر www-data هستیم. اگه خواستی داخل container بشینی و دستورات رو مستقیم بزنی:

$ docker compose exec php bash

بعدش می‌تونی مستقیم دستور بدی (دیگه نیازی به docker compose exec php نیست). برای خارج شدن: exit بزن.

🔧 قدم ۶: chmod vs chown — چه فرقی دارن؟

خیلی از مبتدی‌ها این دو تا رو قاطی می‌کنن. بیا واضح بگم:

🔑 chmod👤 chown
چی رو تغییر می‌ده؟ دسترسی (permission) — rwx چی باشه مالکیت (ownership) — فایل مال کیه
مثال chmod 644 file.txt chown www-data:www-data file.txt
معنی به فایل می‌گه: "اجازه بده owner بتونه بخونه/بنویسه، بقیه فقط بخونن" به فایل می‌گه: "از این به بعد صاحب‌ت باشه www-data و گروه‌ت باشه www-data"
تأثیر روی چه کسایی؟ همه کاربران (بر اساس نقشی که دارن) فقط صاحب جدید فایل
کی استفاده می‌کنیم؟ برای رفع ارور Permission denied برای تغییر مالک فایل به کاربر دیگه

🔍 توی خروجی ls -la کدوم کدومه؟

بیا دوباره به خروجی نگاه کنیم:

-rw-r--r-- 1 www-data www-data 1035 Jul 5 06:43 property-card.blade.php ^^^^^^^^^^^^^ ^^^^^^^^^ ^^^^^^^^^ chmod (اعداد) chown (owner:group)

ستون اول = chmod · ستون سوم و چهارم = chown

🔗 این با Dockerfile خودمون چه ربطی داره؟

توی docker/php/Dockerfile این خط رو داریم:

usermod -u $UID www-data && groupmod -g $GID www-data

این یعنی: user ای به اسم www-data رو بگیر، UID (شناسه عددی) ش رو بکن همون عددی که کاربر ویندوزمون داره (معمولاً ۱۰۰۰).

چرا این کار رو می‌کنیم؟ چون:

  • همه فایل‌های پروژه روی host (ویندوز) توسط کاربر شما ساخته می‌شن
  • اگه UID توی container با host یکی باشه، فایل‌هایی که container می‌سازه (مثل cache توی storage/) روی ویندوز هم قابل نوشتن هستن
  • اگه این کار رو نکنیم، توی ویندوز فایل‌ها رو می‌بینی ولی می‌گه "اجازه دسترسی نداری"

🎭 قدم ۷: umask — چرا فایل جدید با ۶۴۴ ساخته می‌شه نه ۷۷۷؟

تا حالا به این فکر کردی که چرا وقتی یه فایل جدید می‌سازی، خودش permission 644 رو می‌گیره؟ این به خاطر umask هست.

umask = یک ماسک (نقاب) که از permission کامل کم می‌شه تا permission نهایی به دست بیاد

اول ببینیم umask container ما چنده:

$ docker compose exec php sh -c "umask"
0022

خروجی 0022 رو گرفتیم. حالا فرمول محاسبه:

# برای فایل: ۶۶۶ (پیش‌فرض لینوکس برای فایل — چون x نمی‌خواد) - ۰۲۲ (umask ما) ──── ۶۴۴ ✅ (همون rw-r--r--) # برای دایرکتوری: ۷۷۷ (پیش‌فرض لینوکس برای دایرکتوری) - ۰۲۲ (umask ما) ──── ۷۵۵ ✅ (همون rwxr-xr-x)

📌 چرا فایل از ۶۶۶ شروع می‌کنه ولی دایرکتوری از ۷۷۷؟

چون فایل‌های معمولی (مثل PHP, CSS, JS) نباید پیش‌فرض x (اجرا) داشته باشن. اگه هر فایلی که می‌سازی خودش x بگیره، امنیت سیستم به خطر می‌افته. برای همینه که لینوکس فایل رو با ۶۶۶ شروع می‌کنه (بدون x). ولی دایرکتوری برای اینکه بتونی واردش بشی نیاز به x داره، پس با ۷۷۷ شروع می‌کنه.

umask های مختلف چه نتیجه‌ای دارن؟

umaskنتیجه فایلنتیجه دایرکتوریکجا استفاده می‌شه
022644755پیش‌فرض اکثر سیستم‌ها — امن و استاندارد
002664775محیط‌های تیمی — گروه هم می‌تونه بنویسه
077600700محیط‌های امن — فقط خودت دسترسی داری

⚠️ قدم ۸: چرا ۷۷۷ خطرناکه؟ (داستان واقعی)

777 یعنی rwxrwxrwx — هر کسی روی کره زمین می‌تونه:

  • ✗ فایل رو بخونه (حتی اگه فایل .env با رمز دیتابیس باشه)
  • ✗ فایل رو عوض کنه (ویرایش یا پاک کنه)
  • ✗ فایل رو اجرا کنه (اگه script یا PHP باشه)

🎬 سناریوی واقعی که خیلی پیش میاد:

  1. کاربر A یه فرم آپلود توی سایت می‌ذاره که کاربرا بتونن عکس آپلود کنن
  2. یه هکر به جای عکس، یه فایل PHP مخرب (webshell) آپلود می‌کنه
  3. اگه دایرکتوری uploads permission 777 داشته باشه، هکر می‌تونه فایلش رو اجرا کنه
  4. حالا هکر از راه دور می‌تونه کل سرور رو کنترل کنه
# هکر این رو توی مرورگرش می‌زنه: https://your-site.com/uploads/shell.php?cmd=rm -rf / # و کل سایتت پاک می‌شه! 😱

✅ قانون طلایی امنیت: همیشه از کمترین دسترسی لازم استفاده کن (Principle of Least Privilege).

یعنی:

  • فایل معمولی → ۶۴۴
  • دایرکتوری → ۷۵۵
  • فایل حساس (کلید SSH, .env) → ۶۰۰
  • هرگز ۷۷۷ نذار!

📋 قدم ۹: جدول جمع‌بندی تمام اعداد رایج

عددrwxچه موقع استفاده می‌شه
400 r-------- امنیتی بالا — فقط owner می‌تونه بخونه، کسی نمی‌تونه تغییر بده یا اجرا کنه
600 rw------- فایل‌های خیلی حساس — کلید SSH خصوصی (id_rsa)، فایل .env
644 rw-r--r-- استاندارد فایل‌ها — PHP, CSS, JS, HTML, عکس، config
755 rwxr-xr-x استاندارد دایرکتوری‌ها — و فایل‌های اجرایی مثل bash script
775 rwxrwxr-x دایرکتوری گروهی — اعضای گروه هم می‌تونن فایل بسازن (چند نفره روی یه پروژه کار می‌کنن)
777 rwxrwxrwx ⚠️ هرگز! مگه برای یه لحظه تست و بعد برگردون.

✍️ ۵ نکته برای مرور سریع (روی کاغذ بنویس)

  1. ۱. سه بلوک rwx مال کی هستن؟

    هر فایل سه بلوک ۳ تایی داره: rwx برای Owner (صاحب فایل)، rwx برای Group (گروه)، rwx برای Others (بقیه). به علاوه یه کاراکتر اول که نوع فایل رو نشون می‌ده (- فایل، d دایرکتوری).

    -rwxrwxrwx نوع owner group others

  2. ۲. چرا ۷۵۵ برای دایرکتوری‌ها ولی ۶۴۴ برای فایل‌ها؟

    دایرکتوری برای اینکه بتونی cd کنی (وارد بشی) نیاز به x داره. فایل معمولی نیازی به اجرا نداره. پس: فایل ← ۶۴۴ (rw-r--r--) · دایرکتوری ← ۷۵۵ (rwxr-xr-x). هر دو به بقیه اجازه خوندن می‌دن ولی فقط owner می‌تونه تغییر بده.

  3. ۳. chown چیکار می‌کنه؟ (تفاوتش با chmod)

    chmod = دسترسی رو عوض می‌کنه (اعداد rwx). مثل کلید در خونه — می‌گی کی بتونه بیاد تو.
    chown = مالکیت رو عوض می‌کنه (کدوم کاربر صاحب فایله). مثل سند خونه — می‌گی خونه مال کیه.
    مثال: chmod 644 file ← دسترسی ۶۴۴ · chown ali:admins file ← مالک بشه ali و گروه بشه admins.

  4. ۴. umask یعنی چی؟

    یه ماسک پیش‌فرض که موقع ساخت فایل جدید، از ۶۶۶ (فایل) یا ۷۷۷ (دایرکتوری) کم می‌شه تا permission نهایی به دست بیاد. مقدار ۰۲۲ توی container ما باعث میشه فایل‌ها با ۶۴۴ و دایرکتوری‌ها با ۷۵۵ ساخته بشن. اگه umask رو بذاری ۰۷۷، فایل‌های جدید با ۶۰۰ ساخته می‌شن (فقط صاحبش دسترسی داره).

  5. ۵. چرا ۷۷۷ معمولاً بده؟

    چون به هر کسی اجازه هر کاری رو می‌ده. اگه یه هacker بتونه یه فایل توی سرور بذاره (مثلاً از طریق فرم آپلود)، با ۷۷۷ می‌تونه اجراش کنه و کل سرور رو بزنه. همیشه از کمترین دسترسی لازم استفاده کن — فایل ۶۴۴، دایرکتوری ۷۵۵، فایل حساس ۶۰۰. هیچ وقت ۷۷۷ نذار!
